# Docs Update

ドキュメントを最新の変更に合わせて正確に更新する。

## 目的

- 変更内容を正しく反映する。
- 読者が迷わないように手順と要点を整理する。
- 既存の書き方や規約に合わせて一貫性を保つ。

## 対象外

- 軽微な表記修正のみで、更新方針の整理が不要な場合。

## レビュー・改善サイクル（必須）

- 生成・修正する内容（計画、手順、文面、設定、変更案など）は、提示または適用前にレビューする。
- レビュー結果を反映して改善し、これを3回繰り返す。
- 3回目の改善結果を最終案として採用し、ユーザーには最終案のみ提示する（途中案は原則提示しない）。
- 途中案や各ラウンドの内容は、ユーザーから明示的に要求された場合にのみ提示する。
- ユーザー指定や既存規約がある場合は、それらを優先して最終案に反映する。

## 前提確認

- 更新対象（README・CONTRIBUTING・運用ドキュメント・その他）を確認する。
- 変更内容と対象読者を確認する。
- 既存の文体・用語・フォーマットを確認する。
- 変更元の情報（実装差分、仕様、Issue・PR）を確認する。
- 不明点は最小限の質問で補う。

## 情報源の優先順位

- 仕様・実装事実（コード/設定）を最優先する。
- 次に公式運用ドキュメント（README/CONTRIBUTING）を参照する。
- Issue/PRコメントは補助情報として扱い、事実と矛盾する場合は採用しない。
- 判断不能な場合は「未確定事項」として明示し、断定しない。

## 手順

### 1) 変更内容の把握

- 実装差分や仕様変更を整理する。
- 影響範囲とユーザーへの影響を把握する。
- 対象読者（開発者・利用者・運用者）を意識する。

### 2) 反映方針の決定

- 既存の構成を崩さないように差し込む。
- 既存の規約（表記・見出し・テンプレ）に合わせる。

### 3) 更新

- 追加・変更・削除を最小限の差分で行う。
- 手順はステップ順に整理する。
- 例や注意点が必要なら簡潔に追加する。

### 4) 整合性確認

- 他のドキュメントと矛盾していないか確認する。
- リンク、コマンド、パスが正しいか確認する。
- リンク先やコマンドが実際に有効か確認する。
- 用語と見出しレベルを既存文書と揃える。

## 品質チェックリスト

- [ ] 変更根拠が実装事実と一致している
- [ ] 対象読者に必要な手順が不足なく記載されている
- [ ] リンク/コマンド/パスが有効である
- [ ] リリースノート領域（CHANGELOG/バージョン判断）を混在させていない
- [ ] 未確定事項を断定表現で書いていない

## 出力フォーマット

```markdown
# ドキュメント更新: [対象]

## 変更内容

- [要約]

## 反映箇所

- [ファイル・セクション]

## 更新方針

- [方針]

## 追加・変更・削除

- 追加: [内容]
- 変更: [内容]
- 削除: [内容]

## 未確定事項

- [確認待ちの項目]
```

## 返答のガイド

- 変更点と反映先を簡潔に報告する。
- 影響がある手順や設定変更は明確に書く。
- 実行が必要なコマンドがあれば明示する。
- 不明点があれば短く確認する。
- 未確定事項は「確認待ち」として明示する。
