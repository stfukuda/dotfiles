# Env Setup

開発環境のセットアップを再現可能な手順として整備する。

## 目的

- 必須の前提条件を明確化する。
- 再現性の高い手順を短くまとめる。
- 既存の規約・構成に合わせて一貫性を保つ。

## 対象外

- 既に十分な手順があり、軽微な修正だけで済む場合。

## 前提確認

- 対象OS・ランタイム・ツールチェーンを確認する。
- セットアップ対象（開発・CI・本番模擬など）を確認する。
- 既存の手順・テンプレ・規約があれば優先する。
- 必要な権限・ネットワーク要件・依存サービスを確認する。
- 再現性のための前提（権限、ネットワーク制限、必要なアカウント）を明示する。
- 不明点は最小限の質問で補う。

## 手順

### 1) 前提条件の整理

- 必要なOS/バージョン、言語/ランタイム、パッケージ管理ツールを列挙する。
- 必要な外部サービス（DB・API・認証）を整理する。
- 依存関係やバージョン固定の有無を明確にする。
- OS差分や代替手順がある場合は分岐を明示する。

### 2) 手順の構成

- セットアップの順序を決める（例: 依存インストール → 設定 → 初期化）。
- 1ステップ1アクションで書く。
- 失敗時の原因切り分けができる粒度にする。
- 省略できない前提がある場合は手順より前に明記する。

### 3) セットアップ手順の記述

- そのまま実行できるコマンドを提示する。
- 重要な環境変数や設定ファイルを明記する。
- 秘密情報は記載せず、設定方法のみ示す。
- 実行順が重要な場合は理由を添える。

### 4) 検証

- セットアップ完了の確認方法を明示する。
- 実行可能なテストや起動確認を提示する。
- 実行できない場合は代替確認を示す。
- 初回起動時の注意点があれば明記する。

## 避けるべきこと

- 手順の前提が暗黙のまま進む。
- コマンドの前提（権限・OS差分）が不明確。
- 秘密情報を本文に含める。
- 手順が冗長で再現性が落ちること。

## 出力フォーマット

```markdown
# セットアップ手順: [対象]

## 前提条件

- OS:
- ランタイム/言語:
- ツールチェーン:
- 依存サービス:
- 権限/ネットワーク:

## 手順

1. [手順]
2. [手順]

## 設定

- 環境変数:
- 設定ファイル:
- 秘密情報の設定方法:

## 検証

- 確認方法:
- 代替確認:
- 初回起動時の注意:
```

## 返答のガイド

- 手順は実行順に簡潔に書く。
- 不足情報があれば短く確認する。
- セットアップ完了の確認方法を必ず書く。
